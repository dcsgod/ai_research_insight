The problem

AI research moves faster than any human can track

We spent three months manually scanning arXiv every morning, watching GitHub trending, skimming HuggingFace releases — and still felt like we were always a week behind the curve. So we automated everything.

It started as a personal frustration. The AI ecosystem publishes roughly 200 new papers per day on arXiv alone. Add GitHub repositories, HuggingFace model releases, PapersWithCode implementations, and Reddit community discussions — and you're looking at a signal volume that's genuinely impossible to process manually.

The existing solutions weren't cutting it. RSS feeds give you chronological dumps with no quality signal. Newsletters curate last week's news. Twitter/X is excellent for virality but terrible for discovering slow-burn research that compounds into something important. What we wanted was something closer to how a quantitative trader thinks about markets: continuous ingestion, algorithmic scoring, and momentum forecasting.

We wanted to know not just what was trending right now — but what was going to be trending in seven days. That predictive layer is what separates intelligence from aggregation.

Why not just use an LLM to summarise everything?

We tried that first. The problem is that a language model gives you a snapshot — it can synthesise what's in its context window beautifully, but it has no concept of velocity. It can't tell you that GraphRAG was getting 50 GitHub stars per day last month and is now getting 2,000. That momentum signal is what matters most for early trend detection, and it requires a real time-series pipeline, not a language model.

LLMs play a supporting role in this system — generating natural-language insight summaries once the ML pipeline has already done the heavy lifting of scoring and ranking. The intelligence is in the infrastructure, not the chatbot.

What we built

A live intelligence dashboard, not a glorified RSS reader

The frontend is a Next.js 15 dashboard with six views: a main trend feed, a papers explorer, a GitHub repos tracker, a forecasts page, a topic graph, and a live WebSocket feed that streams new ingestion events in real time. Think Bloomberg Terminal aesthetics meets research tooling.

The live feed is powered by WebSocket subscriptions to a Redis Streams backend. Every time the ingestion orchestrator processes a new item and it scores above a quality threshold, it publishes to the stream. The frontend subscribes and appends items in real time — no polling, no page refreshes.

We deliberately kept the UI dense. AI practitioners are comfortable with information density — this isn't a consumer app, it's a professional tool. The aesthetic is intentionally terminal-like in the dark panels and data-forward in the light sections.

Data pipeline

Five MCP servers, one ingestion orchestrator

The data layer is built around Anthropic's Model Context Protocol (MCP) — each data source gets its own MCP server that handles auth, rate-limiting, pagination, and schema normalisation. The ingestion orchestrator treats them all identically at the transport layer.

📄 arXiv Latest preprints, abstracts, author graphs
⬡ GitHub Star velocity, forks, commit cadence
🤗 HuggingFace Model releases, dataset cards, downloads
💡 PapersWCode SOTA benchmarks, code implementations
💬 Reddit r/MachineLearning community sentiment

The choice of MCP wasn't incidental. We evaluated plain REST adapters and a few message-queue patterns before landing on MCP. The key advantage is that each server declares its capabilities, data schema, and rate-limit budget in a standard way — the orchestrator doesn't need to know anything source-specific. Adding a new source (Semantic Scholar, ACL Anthology, Elsevier) means writing one new MCP server and plugging it in. No changes to the ranking engine or frontend.

What happens when a source goes down?

Each MCP server has an exponential backoff retry policy and writes to a dead-letter queue in Redis if it fails after three attempts. The orchestrator tracks source health independently, so a Reddit API outage doesn't stall arXiv ingestion. Kafka handles the event streaming between the MCP layer and the backend — we chose Kafka over RabbitMQ for its durability guarantees and the ability to replay ingestion events for backfill.

# Run ingestion manually — useful for backfill or testing python scripts/run_ingestion.py --sources all # Ingest only specific sources python scripts/run_ingestion.py --sources arxiv github # Available sources: arxiv, github, huggingface, paperswithcode, reddit
Ranking engine

The X-style ranking formula — and why we built it this way

When X (formerly Twitter) surfaced its ranking algorithm, a few things became clear: the best feed algorithms are composites of independent signals, each measuring something meaningfully different. Citation count alone doesn't tell you momentum. GitHub stars alone doesn't tell you research quality. You need both — weighted by what matters for the use case.

Our ranking formula uses five signals. The weights were tuned empirically by running the algorithm on six months of historical data and comparing its top-10 outputs against what the community retrospectively agreed were the most important papers and repos of each week.

trend_score =
  0.35 × growth_velocity  // how fast is attention accelerating?
  + 0.25 × github_activity  // is anyone actually building with it?
  + 0.20 × citation_acceleration  // are researchers citing it?
  + 0.10 × community_engagement  // is the community talking?
  + 0.10 × novelty_score  // is this genuinely new?

Growth velocity gets the highest weight because it's the best leading indicator we found. A paper that doubles its daily citation count over three consecutive days is almost always genuinely important — the field is noticing it. GitHub activity is a strong proxy for practical applicability: researchers star things they find interesting, but they fork and commit to things they're actually building with.

The novelty score deserves a special mention. It's computed by embedding each item with SentenceTransformers and checking its cosine distance from the existing top-50 items in Qdrant. If something is semantically very close to what's already ranked highly, it gets a novelty penalty. This prevents the feed from being dominated by five variations of the same paper — we call this diversity injection, and it significantly improved perceived feed quality in user testing.

The multi-stage pipeline

The ranking doesn't happen in a single pass. We start with 500 candidates and run them through a sequence of filters and boosters before landing on Top-K results.

01 Candidate Retrieval N = 500 items from all sources PostgreSQL
02 Signal Filtering Quality threshold — remove spam & duplicates FastAPI
03 Embedding Similarity Semantic relevance via vector search Qdrant
04 Momentum Scoring Prophet time-series forecast integration Prophet
05 Engagement Boosting Recency decay + virality multiplier Scorer
06 Novelty Injection Semantic diversity enforcement BERTopic
07 Re-Ranking Final pass with XGBoost learned ranker XGBoost
08 Top-K Results Served via REST API + WebSocket stream API
ML pipeline

BERTopic, Prophet, and why we chose them over fancier alternatives

When people hear "ML pipeline" they often expect something exotic. Ours is deliberately practical. BERTopic for topic modelling. Facebook Prophet for time-series forecasting. XGBoost for learning-to-rank. All of these are mature, well-understood tools with sensible defaults and excellent documentation.

We evaluated more complex options. We looked at fine-tuning a custom transformer for topic extraction, and we looked at temporal fusion transformers for forecasting. Both outperformed our choices in benchmark conditions and both were significantly harder to operate in production. BERTopic with a decent embedding model gets 90% of the quality at 20% of the infrastructure complexity.

Topic extraction with BERTopic

Every ingested item — paper abstract, repo description, Reddit post — gets embedded with sentence-transformers/all-mpnet-base-v2 and passed through BERTopic. The model maintains a live topic graph that updates incrementally as new items arrive. Topics are labelled automatically using the top-n words per cluster, but we also run a quick Qwen 2.5 inference step to generate human-readable topic names like "Graph-based RAG for enterprise retrieval" rather than "graph retrieval augmented generation enterprise".

Forecasting with Prophet

Each tracked topic gets a Prophet model trained on its 90-day engagement history. We forecast 7 days ahead, updating models every 6 hours. Prophet handles the seasonality patterns well — AI Twitter tends to spike Monday–Wednesday, cool off Thursday–Friday, and go quiet on weekends. Accounting for that weekly seasonality significantly improved forecast accuracy.

AI-generated insight · Qwen 2.5 via Ollama · live output

"GraphRAG is rapidly gaining momentum due to enterprise retrieval applications and multi-agent workflows. Star growth has accelerated 340% in the past 30 days, with 12 new implementations released this week. Forecast: top-3 trending topic in 7 days with 87% confidence."

The LLM insight generation runs asynchronously — it doesn't block the ranking pipeline. Qwen 2.5 7B via Ollama gives us good enough quality for this task on a single GPU, and being open-source means we're not paying per-token for something that runs continuously. For teams without GPU access, any OpenAI-compatible endpoint (including the Anthropic API) can be substituted by changing one environment variable.

PostgreSQL stores the canonical records — papers, repos, models, their metadata and time-series signal history. Qdrant stores the vector embeddings and powers semantic search and similarity queries. Redis serves two purposes: caching expensive ranking computations (TTL: 5 minutes) and powering the Streams-based event bus between the ingestion layer and the API. Kafka handles higher-volume event streaming between MCP servers and the backend.

Nginx sits in front of everything as a reverse proxy, routing /api/* to FastAPI and everything else to Next.js. The WebSocket endpoint at /ws/live-feed is proxied with upgraded connections. All services run in Docker Compose — the entire stack starts with a single command.

Technology choices

The full tech stack — and the reasoning behind each choice

We're not neutral on these choices. Every technology here was picked after evaluating alternatives, and some of the tradeoffs are worth explaining.

FastAPI over Django or Flask because we needed async I/O for the WebSocket layer and the ingestion pipeline runs a lot of concurrent network requests. Pydantic v2 is a significant performance improvement over v1 and the schema validation is critical when you're ingesting from five heterogeneous sources. ECharts over Recharts or Victory because it handles large time-series datasets significantly better out of the box.

No OpenAI dependency

The entire ML pipeline runs locally. BERTopic, SentenceTransformers, Prophet, XGBoost, and Qwen 2.5 via Ollama — all self-hosted. The only external APIs are the data sources themselves (arXiv is free, GitHub needs a PAT, Reddit needs an app key). Zero per-token costs at inference time.

The backend exposes a clean REST API documented automatically via FastAPI's OpenAPI integration. Everything available in the dashboard is available via API — useful for building integrations, alerting pipelines, or custom frontends on top of the ranking engine.

What's next

Where we're taking this

The core platform is stable and running in production for our internal team. The next major piece is personalised trend feeds — learning individual reading patterns and adjusting the novelty_score and community_engagement weights per user. Someone who focuses on computer vision has different signal priorities than someone who cares primarily about reasoning systems.

We're also working on citation graph integration with Semantic Scholar's API. Right now citation_acceleration is computed from PapersWithCode and arXiv metadata — adding the full Semantic Scholar citation graph would make this signal significantly richer, especially for papers in the first 1–2 weeks after publication.

Finally, email digests and Slack alerts are the most-requested feature. The ranking engine already runs continuously — it's just a matter of building the notification layer on top. Subscribe to a topic cluster, set a score threshold, and get a Slack message when something spikes.

If you're interested in contributing, the repo is MIT-licensed and the architecture is deliberately modular. Adding a new MCP server for a new data source is the easiest starting point — you'll find the existing servers in mcp_servers/ are each under 200 lines.

TL;DR

We built the intelligence layer that was missing between raw arXiv feeds and curated newsletters. The stack is production-grade, fully self-hostable, and open source. docker-compose up -d and you're running.
